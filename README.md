# Expo Android cold-start build failure — Gradle 9.0.0 `Could not read workspace metadata`

Minimal reproduction for a bug in Expo SDK 55 / React Native 0.83.2 where a clean Android build fails on a cold Gradle cache.

## Environment

- Expo SDK: 55.0.4
- React Native: 0.83.2
- `@react-native/gradle-plugin`: 0.83.2
- Gradle (as generated by `expo prebuild`): **9.0.0**
- Node: 22
- Java: 17
- OS: macOS 15 / Ubuntu (GitHub Actions)

## Steps to reproduce

```bash
# 1. Clone and install
git clone <this-repo>
cd expo-bug-repro
npm install

# 2. Simulate a fresh machine by clearing the Gradle cache
rm -rf ~/.gradle/caches

# 3. Build Android
./android/gradlew app:assembleDebug
```

The CI workflow (`.github/workflows/repro.yml`) reproduces this automatically — GitHub Actions runners start with an empty `~/.gradle/caches`.

## Expected behaviour

Build succeeds. Gradle generates its kotlin-dsl accessor cache on first run.

## Actual behaviour

```
FAILURE: Build failed with an exception.

* Where:
Settings file '.../android/settings.gradle' line: 21

* What went wrong:
Error resolving plugin [id: 'com.facebook.react.settings']
> Could not read workspace metadata from
  ~/.gradle/caches/9.0.0/kotlin-dsl/accessors/<hash>/metadata.bin

Caused by: java.io.FileNotFoundException: ...metadata.bin (No such file or directory)
    at org.gradle.internal.execution.history.impl.DefaultImmutableWorkspaceMetadataStore.loadWorkspaceMetadata
```

## Root cause

`expo prebuild` generates an `android/settings.gradle` that includes two composite builds:

```groovy
pluginManagement {
  // 1. @react-native/gradle-plugin — resolved dynamically via providers.exec + node
  def reactNativeGradlePlugin = new File(
    providers.exec {
      commandLine("node", "--print", "require.resolve('@react-native/gradle-plugin/...')")
    }.standardOutput.asText.get().trim()
  ).getParentFile().absolutePath
  includeBuild(reactNativeGradlePlugin)

  // 2. expo-gradle-plugin — second composite build
  def expoPluginsPath = new File(...).absolutePath
  includeBuild(expoPluginsPath)
}
plugins {
  id("com.facebook.react.settings")
  id("expo-autolinking-settings")   // from the second included build
}
```

This dual-composite-build structure, combined with `providers.exec` for dynamic path resolution at configuration time, triggers a bug in Gradle 9.0.0's `DefaultImmutableWorkspaceMetadataStore`: on a cold cache, it attempts to *read* the kotlin-dsl accessor metadata before *writing* it, throwing `FileNotFoundException`.

**This does not affect vanilla React Native projects**, which use a simpler `settings.gradle` with a single `includeBuild` and no `providers.exec`. Confirmed by running the [react-native-community reproducer template](https://github.com/react-native-community/reproducer-react-native) with RN 0.84.1 — builds successfully on cold cache.

The bug is fixed in Gradle 9.1.0. Since `expo prebuild` pins `gradle-9.0.0-bin.zip` in `gradle-wrapper.properties`, every developer or CI runner with an empty `~/.gradle/caches` is affected on first build.

## Workaround

After running `expo prebuild`, patch `android/gradle/wrapper/gradle-wrapper.properties`:

```
distributionUrl=https\://services.gradle.org/distributions/gradle-9.3.1-bin.zip
```

## Fix

Update the Gradle wrapper version generated by `expo prebuild` from `9.0.0` to `9.1.0` or later.
